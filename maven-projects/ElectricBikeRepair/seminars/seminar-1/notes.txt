Thrown exceptions: Don't catch it in the controller — let it propagate to the View, and handle it there. The View is where user interaction happens, so it's the right place to display error messages.
Controller/Model - throw exceptions to signal problems (which you already do)
View - catch exceptions and present user-friendly messages
Don't catch exceptions in the middle layers unless you can actually recover from them

Exception handlings:
IllegalArgumentException — "you gave me bad input". The caller passed an invalid argument. The fix is on the caller's side.
Example: invalid phone number, customer not found for given phone number
IllegalStateException — "something is wrong internally". The object/system is in a state that shouldn't be possible. The fix is in the code itself.
Example: an ID exists in one map but not another (data inconsistency), or a method is called in the wrong order
In your project:
User types a wrong phone number → IllegalArgumentException (user's fault, View catches and shows error)
Registry has an ID but no matching object → IllegalStateException (code bug, should be investigated)

User wrong behavior (should be IllegalArgumentException, caught in View):
* TelephoneNumber — invalid phone format ✅ already correct
* Controller.findRepairOrder — no repair order for customer ✅ already correct
* ReceptionistController.createRepairOrder — customer not found. Currently IllegalStateException, but this is user input error → should be IllegalArgumentException

Code bug (should be NullPointerException or IllegalStateException, propagated):
* TechnicianController.createDiagnosticReport — findByRepairOrderId returns null, then repairOrder.setDiagnosticReportDTO throws NPE. This is a code bug (invalid ID passed internally).
* TechnicianController.createProposedRepairTask — same issue
* ReceptionistController.acceptOrder — same issue
* ReceptionistController.rejectOrder — same issue

IllegalArgumentException — user wrong behavior (caught in View):
* TelephoneNumber — invalid phone number format
* ReceptionistController.createRepairOrder — customer not found for given phone number
* Controller.findRepairOrder — no repair order for customer

IllegalStateException — code bug (propagated, should never happen in correct flow):
* ReceptionistController.acceptOrder/rejectOrder — repair order ID doesn't exist
* TechnicianController.createDiagnosticReport/createProposedRepairTask — repair order ID doesn't exist

The View already catches both IllegalArgumentException | IllegalStateException, so if an IllegalStateException does occur, it will still show an error message rather than crashing — but it signals a programming error that should be investigated.
