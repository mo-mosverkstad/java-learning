DTO-policy
1. DBHandler/Integration layer -> immutable during program -> directly as DTO (place in DBHandler/integration layer) e.g. customer and bike
2. Changed during program -> create an immutable class in model layer e.g. RepairOrder
3. Traversing from controller to view changes the entity class into a immutable DTO
(Access security as primary target rather than performance)
4. CreateDiagnosticReport, createProposedRepairTask takes DTO as argument, since contents of report and proposed repair tasks may change
5. CreateRepairOrder takes argument ProblemDTO (with problemDescription and brokenBike). May include more attributes in future e.g. urgency, extra tasks

Controller should be divided into two controllers: ReceptionistController and TechnicianController due to different authorization (a technician cannot deny a repair order by himself). Both controller classes are derived from an abstract base class called AbstractController or simply Controller since both authorization roles share same principles.

Data read from JSON file. Data and code logic should be separated. (Encapsulation)

Record usage as DTO since it is immutable. RepairOrderDTO and RepairOrder share logic (toString) but the DTO version has final.
Record / Immutable DTO + Mutable Model — Keep RepairOrderDTO as a Java record (or immutable class with final fields) for data transfer, and RepairOrder as the mutable domain model. Use a factory method or converter to map between them .toDTO
The solution is to extract a static method that does not require an object instance (toString method since it is redundant and repeated in the two classes). The visibility modifier is default so other directory than model cannot access it.


Thrown exceptions: Don't catch it in the controller — let it propagate to the View, and handle it there. The View is where user interaction happens, so it's the right place to display error messages.
Controller/Model - throw exceptions to signal problems (which you already do)
View - catch exceptions and present user-friendly messages
Don't catch exceptions in the middle layers unless you can actually recover from them
